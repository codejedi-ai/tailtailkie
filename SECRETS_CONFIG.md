# Secrets Configuration Guide

This document explains how secrets are configured and loaded in TensorStore.

## Secret Flow

1. **`.env` file** - Your source of truth (edit this file)
2. **`docker-secrets.env`** - Generated from `.env` (auto-generated by setup.sh)
3. **Docker Compose** - Loads secrets via `env_file` and `environment` sections

## Required Secrets

### Critical (Must Set)
- `CLERK_SECRET_KEY` - Clerk authentication secret key (starts with `sk_test_` or `sk_live_`)
- `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` - Clerk publishable key (starts with `pk_test_` or `pk_live_`)
- `MONGODB_ROOT_PASSWORD` - MongoDB root password (change from default `changeme`)

### Optional (Have Defaults)
- `MILVUS_TOKEN` - Milvus authentication token (optional for local Milvus)
- `MILVUS_USER` - Milvus username (optional)
- `MINIO_ACCESS_KEY` - MinIO access key (default: `minioadmin`)
- `MINIO_SECRET_KEY` - MinIO secret key (default: `minioadmin`)

## How Secrets Are Loaded

### Backend Service
Secrets are loaded in this order (later overrides earlier):
1. `env_file: docker-secrets.env` - Loads all secrets from file
2. `environment:` section - Explicitly sets variables (uses shell substitution)

**Important**: The `environment` section uses `${VAR:-default}` syntax which:
- Reads from the shell environment where `docker-compose` runs
- Falls back to the default value if not found
- The `env_file` values are loaded into the container, but shell substitution happens before that

### Frontend Service
Same pattern - loads from `docker-secrets.env` first, then `environment` section.

## Verification

Run the verification script to check your secrets:

```bash
./scripts/verify-secrets.sh
```

This will:
- ✅ Check if `.env` exists
- ✅ Validate required secrets are set
- ✅ Check if `docker-secrets.env` is generated
- ✅ Verify secrets are not using placeholder values

## Common Issues

### Issue: Secrets not loading
**Problem**: Docker Compose uses shell variable substitution `${VAR}` which reads from your shell environment, not from `env_file`.

**Solution**: 
1. Make sure you run `./scripts/setup.sh` which loads `.env` and creates `docker-secrets.env`
2. The `env_file` loads secrets into the container environment
3. For shell substitution in docker-compose.yml, you may need to export variables or use a different approach

### Issue: Default values being used
**Problem**: If a secret isn't set, docker-compose uses the default value (e.g., `changeme`).

**Solution**: 
1. Edit `.env` file with actual values
2. Run `./scripts/setup.sh` to regenerate `docker-secrets.env`
3. Verify with `./scripts/verify-secrets.sh`

### Issue: Placeholder values
**Problem**: Secrets still have placeholder values like `sk_test_...`

**Solution**:
1. Edit `.env` file
2. Replace placeholders with actual values from your Clerk dashboard
3. Run `./scripts/setup.sh` again

## Environment Variables Used

### Backend Needs:
- `MONGODB_URI` or `DATABASE_URL` - MongoDB connection string
- `MILVUS_URI` - Milvus instance URI
- `MILVUS_TOKEN` - Milvus authentication token
- `CLERK_SECRET_KEY` - For JWT verification
- `CLERK_FRONTEND_API` - Clerk API URL (default: https://api.clerk.com)

### Frontend Needs:
- `FLASK_API_URL` - Backend API URL (internal)
- `NEXT_PUBLIC_FLASK_API_URL` - Backend API URL (for client-side)
- `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` - Clerk public key
- `CLERK_SECRET_KEY` - For server-side operations

## Best Practices

1. **Never commit `.env` or `docker-secrets.env`** - They're in `.gitignore`
2. **Use strong passwords** - Especially for `MONGODB_ROOT_PASSWORD`
3. **Rotate secrets regularly** - Especially in production
4. **Use different values for dev/staging/prod**
5. **Verify before starting** - Always run `./scripts/verify-secrets.sh` first

## Quick Fix

If secrets aren't working:

```bash
# 1. Edit .env with real values
nano .env

# 2. Regenerate docker-secrets.env
./scripts/setup.sh

# 3. Verify
./scripts/verify-secrets.sh

# 4. Start services
./scripts/start.sh
```
